PDF → Markdown: Model × Harness Benchmark

How model capability and harness design interact — a 3-model × 2-harness natural experiment converting MASQUE protocol articles to clean Markdown.

Setup

Two PDF articles were converted to Markdown with YAML frontmatter using the same prompt across three models and two AI coding harnesses. The PDFs were browser-printed web pages — both carry repeating page headers, footer chrome, and page-number stamps. The InstaTunnel article additionally uses an OpenType ligature font set (fi, fl, ffi) that some extraction tools fail to decode.

Models tested: Claude Sonnet 4.6 (vision), Kimi K2.6 (vision), DeepSeek-V4-Pro (text-only).

Finding 1: The Harness Effect

The most striking result: Kimi K2.6 through Claude Code produced output nearly indistinguishable from the Sonnet runs; Kimi K2.6 through OpenCode produced the second-worst output in the set. Same model, same task, same source documents. The harness is the variable.

The Claude Code system prompt shapes the model toward editorial judgment — "convert to Markdown" means decide what belongs, not transcribe everything. OpenCode's prompt apparently does not carry this signal, and Kimi treated the task mechanically.

Finding 3: OpenCode's Inconsistent Vision Routing

OpenCode sent PDF pages as images to Claude Sonnet 4.6, but when Kimi K2.6 tried to read the PDFs, it received: "ERROR: Cannot read pdf (this model does not support pdf input)". Kimi then installed pymupdf4llm and proceeded with text extraction.

This matters because Kimi K2.6 is vision-capable. OpenCode appears to maintain a model capability registry that does not yet include Kimi — so a capable model was silently downgraded to the text-extraction path.

Model Vision capable? OpenCode PDF routing Outcome
Sonnet 4.6 Yes Images + extracted text (dual signal) Correct
Kimi K2.6 Yes Error → pymupdf4llm fallback Misconfigured — capable model downgraded
DeepSeek-V4-Pro No Raw text extraction Correct (nothing better was possible)

Notably, OpenCode gave Sonnet both rendered images and extracted text simultaneously — richer input than Claude Code's images-only approach. The extra pdf_file field in Sonnet/OpenCode's frontmatter is a signal that OpenCode passes additional file provenance metadata through its tool layer.

This introduces a third variable beyond model capability and harness behavioral shaping: harness capability detection fidelity. A harness that misclassifies a capable model imposes a quality ceiling that is invisible to the user at task time.

Finding 4: Extraction Library Quality

For models that were not given images, the choice of PDF text extraction library determined whether the raw input was readable or garbled. Three libraries appeared across the runs, plus the baseline of no library at all.

Per-Run Behaviour Summary

Claude Sonnet 4.6 / Claude Code  A

The Read tool rendered each PDF page as an image. The model saw the visual layout directly, enabling correct heading hierarchy, diagram reconstruction (Markdown table), and the stripping of all page chrome. Frontmatter included title, author, publication date, source URL, publisher, source type, and tags. No notable errors.

Claude Sonnet 4.6 / OpenCode  A

OpenCode provided both rendered images and extracted text simultaneously — a dual-signal input richer than Claude Code's images-only approach. Output is essentially identical in quality to the Claude Code run, with minor stylistic differences: YAML tags as a proper list, a pdf_file provenance field added to frontmatter, and slightly more consistent inline code formatting on protocol terms. The HTTP protocol stack diagram was silently omitted (not reconstructed), which is the only meaningful delta from the Claude Code run.

Kimi K2.6 / Claude Code  A−

Despite receiving no visual input, Kimi matched the Sonnet output in almost every measurable dimension. It used pdfplumber (installed in a temporary venv — notably choosing venv over --break-system-packages) and made all the right editorial choices: stripped chrome, synthesized document metadata, correct code blocks, continuous list numbering. The only gaps are the missing diagram (silently omitted) and a two-line header artefact at the top of the InstaTunnel file (InstaTunnel / JUN 06, 2026 not stripped). The harness clearly drove strong editorial judgment from a text-only input.

DeepSeek-V4-Pro / Claude Code  B+

Used PyMuPDF (fitz) directly rather than the higher-level pymupdf4llm wrapper, producing clean text. Made all editorial decisions correctly: stripped chrome, synthesized frontmatter from article content. Uniquely, it also read the PDF metadata dictionary and added pdf_creator: Firefox Developer Edition 152.0 and pdf_producer: cairo 1.18.4 — the most complete provenance information of any run. Errors are structural: without visual layout, it nested The current state of WireGuard as a child of Before the MASQUE and misplaced one paragraph. One factual extraction error: reported HTTP/3 CDN adoption at 72% where the source says 78%.

Kimi K2.6 / OpenCode  C+

Used pymupdf4llm, which handled the Cloudflare PDF's ligatures but failed on the InstaTunnel PDF's font encoding. Page number stamps and repeating URL headers were retained throughout. Frontmatter described file-system location and extraction timestamp, not document identity. One false heading (## it intentionally.) from a page-break fragment. Code block content collapsed to a single line; list numbering reset after the code block. Did strip the Substack subscribe and comment widgets. Better than DeepSeek/OpenCode because ligatures were partly preserved and heading structure was approximately correct.

DeepSeek-V4-Pro / OpenCode  F

The baseline failure mode. Raw binary PDF text extraction produced garbled output on both documents: ligatures decoded as whitespace throughout, ragged line wrapping from PDF column layout preserved verbatim, page headers and footers repeated on every section, and all paragraph text run together without Markdown structure. Frontmatter contained empty title and author fields with only file-level metadata. The model made no recovery attempt — it accepted the degraded input and passed it through. The correct heading structure is the one bright spot: because DeepSeek lacks vision, it had no layout information to misread, so it at least did not introduce structural errors.

Takeaways

1. Harness behavioral shaping is a first-order variable

The Kimi comparison is the cleanest evidence: the same model ranged from C+ to A− depending solely on which harness it ran through. The Claude Code system prompt encodes an implicit contract — "convert" means make editorial decisions, not transcribe — that OpenCode's prompt apparently does not.

2. Vision capability determines a quality ceiling the harness cannot raise

A good harness got DeepSeek-V4-Pro to B+ work. But the structural errors from working with flat extracted text — wrong heading nesting, detached paragraphs — are exactly the kind of thing a visual scan of the source document resolves instantly. The harness can change what the model does with its input; it cannot change what the model's input was.

3. Harness capability detection is a third independent variable

OpenCode silently downgraded a vision-capable model to text extraction because of what appears to be an incomplete model registry. This failure mode is invisible to the user at task time and produces output that looks like a model limitation when it is actually a harness configuration issue.

4. Failure recovery behaviour separates good agents from bad ones

When the PDF read failed, DeepSeek/OpenCode accepted the degraded output; Kimi/OpenCode installed a better library; DeepSeek/Claude Code installed a better library and read the PDF metadata dictionary for additional context. The willingness to diagnose and recover from tool failures — rather than pass garbage input to the model and hope — is a meaningful agentic capability differentiator.

5. Extraction library choice has material impact on output quality

Raw binary extraction is not viable for documents with OpenType ligature fonts. Between the three usable libraries, pdfplumber and PyMuPDF (fitz) are more robust than pymupdf4llm's auto-formatting wrapper, which had a font encoding failure on one of the two source PDFs.

Practical recommendation: For PDF-to-Markdown conversion, prefer a vision-capable model through a harness that correctly routes PDFs as images. If text extraction is unavoidable, prompt explicitly for editorial conversion (not transcription), and choose pdfplumber or PyMuPDF (fitz) over auto-formatting wrappers. Verify harness capability detection before assuming a model's limitations are inherent rather than configuration-imposed.
